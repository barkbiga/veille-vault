## Top découvertes

### 🔥 #1 — MCP est insecure-by-design et Anthropic le assume
**Source :** Infosecurity Magazine / Ox Security | **Date :** 2026-04-15 | **Score :** 5.5

Ox Security a révélé une faille architecturale dans l'interface STDIO de MCP permettant l'exécution de commandes arbitraires — même en cas d'échec du processus. Tous les SDKs officiels d'Anthropic (Python, TypeScript, Java, Rust) sont affectés. La réponse d'Anthropic est le véritable signal : refus de patch, qualification en 'comportement attendu', rejet de responsabilité sur les développeurs.

**Implication immédiate :** Tout déploiement MCP en production financière sans sandboxing explicite constitue un risque DORA non documenté. La conformité Zero Trust ne suffit pas si le protocole sous-jacent est délibérément insécurisé par son éditeur.

**Action :** Auditer les déploiements MCP existants. Exiger un threat model MCP dans tout nouveau cahier des charges agent IA. Documenter le refus de patch Anthropic comme risque tiers ICT (DORA art. 28).

### ⭐ #2 — La draft IETF AIMS : solide sur l'auth, silencieuse sur l'authorization
**Source :** IETF Datatracker (AWS, Zscaler, Ping Identity, Defakto) | **Date :** 2026-03-30 | **Score :** 5.4

Le framework AIMS à 8 couches compose WIMSE, SPIFFE et OAuth 2.0 de manière élégante. Directement applicable aux architectures Zero Trust multi-cloud en services financiers. Mais la section Security Considerations contient deux mots : 'TODO Security'. L'authentification est traitée avec sérieux ; l'autorisation per-action est renvoyée entièrement à l'implémenteur.

**Implication :** Conformité AIMS ≠ conformité réglementaire. Le gap d'autorisation devient une dette architecturale invisible que les équipes projet ont tendance à ne pas documenter.

**Action :** Mapper les 8 couches AIMS sur votre architecture. Documenter le gap d'autorisation per-action comme risque résiduel explicite dans le registre de risques.

### 🎯 #3 — Le gap auth→authz : signal transversal fort (3 sources convergentes)
**Source :** RockCyber Musings | **Date :** 2026-03-17 | **Score :** 5.3

Analyse indépendante qui converge avec la draft IETF et le paper arXiv : l'écosystème règle l'authentification des agents, mais laisse l'autorisation per-action dans un vide complet. Ajoute la dimension réglementaire : le Colorado AI Act (juin 2026) impose un 'reasonable care' sur les systèmes IA à risque élevé — un périmètre qui couvre exactement ce gap non adressé par les standards.

**Signal transversal :** Trois sources indépendantes (IETF, blog indépendant, arXiv) convergent sur le même gap cette semaine. C'est la définition d'un signal fort.

### 📡 #4 — AIP et la Verifiable Intent Mastercard : le signal faible à surveiller
**Source :** arXiv | **Date :** 2026-03-31 | **Score :** 5.2

La cartographie la plus complète des gaps techniques : OAuth 2.1 (tokens opaques), WIMSE (2-parties seulement), Agentic JWT (immutabilité problématique), SCIM agents (lifecycle seulement). Aucune solution existante ne combine délégation atténuable offline + politique chaînée + provenance-aware + bindings MCP/A2A.

**Signal faible :** La 'Verifiable Intent' proposal de Mastercard pour le commerce agent est mentionnée. Premier acteur financier de premier rang à travailler sur ce problème. À surveiller comme signal d'adoption enterprise du concept de délégation vérifiable.

### ⚙️ #5 — Cloudflare : le RBAC resource-scoped comme pattern NHI (avec biais vendor)
**Source :** Cloudflare Blog | **Date :** 2026-04-17 | **Score :** 4.5

Retex d'implémentation à l'échelle sur l'unification identités humaines/non-humaines. Pattern extractible : RBAC étendu aux ressources (et non plus aux seuls rôles) comme building block Zero Trust NHI. Biais vendor réel (Agents Week promotionnelle). Transposabilité aux architectures bancaires on-premise non démontrée.

**Action :** Extraire le pattern RBAC resource-scoped indépendamment du vendor. L'appliquer comme exigence dans les RFP IAM/PAM.
